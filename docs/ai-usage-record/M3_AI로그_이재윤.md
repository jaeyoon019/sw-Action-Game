### 로그 #1

| 항목 | 내용 |
|------|------|
| 날짜 | 2026-04-28 |
| 관련 주차 / 활동 | 6주차 — 요구사항 정의서 마크다운 변환 및 정리 |
| 사용 도구 | Claude (웹) |
| 역할 연관성 | PM 역할로서 분석가가 작성한 요구사항 초안의 형식 통일·문서 정리 진행 |

**프롬프트 (입력한 내용 요약):**
```
첨부한 요구사항정의서(.docx)를 Markdown으로 변환해줘.
- FR/NFR을 표로 분리
- 각 요구사항에 FR-01, NFR-01 형식의 ID 부여
- "~할 수 있어야 한다" 형식으로 문장 통일
- 우선순위 컬럼(상/중/하) 유지
```

**AI 응답 요약:**
> docx의 텍스트를 표 구조로 재구성하여 FR 5개·NFR 2개를 마크다운 표로 변환. ID 부여 및 문장 형식 통일까지 한 번에 처리. 다만 일부 비기능 요구사항을 FR로 잘못 분류한 항목이 있었음.

**비판적 검토:**
> "응답 속도가 빠르다"는 항목을 FR로 분류한 부분이 잘못되어 NFR(성능)로 직접 재분류함. 또한 일부 요구사항이 너무 추상적이라("편리한 UI 제공") 측정 가능한 형태("3클릭 이내 진입")로 수정 필요.

**팀/개인 수정 내용:**
> 분석가(김남준)와 회의 후 NFR 카테고리 재분류, 측정 가능한 표현으로 수정. AI가 만든 표 구조는 유지하되 내용은 팀 합의로 다시 작성.

**최종 활용 결과:**
> `docs/requirements.md`에 반영. AI는 "형식 정리" 단계에서만 기여, 요구사항 내용 자체는 팀 토의로 확정.

**프롬프트:**
```
2D 횡스크롤 액션 게임 팀 프로젝트(5인, 15주차)의 WBS를 작성해줘.
M1 기획(4~8주), M2 설계(9~12주), M3 구현·보고(13~15주) 구조.

**AI 응답 요약:**
> 14개 작업 항목을 단계·담당자·산출물·계획주차 컬럼으로 정리한 표를 제시. 학부 SW공학 과목 표준 일정에 잘 맞춤.

**팀/개인 수정 내용:**
> 프로토타입 작업을 13주 단일 항목에서 분리하지 않고 MVP 스코프 축소로 대응(별도 로그 #6 참조). 코딩 표준은 일정 자체는 14주로 두되, 구현 시 사실상 먼저 적용하도록 운영.

**최종 활용 결과:**
> `docs/wbs.md`에 반영. 이후 5/19, 6/10 실제 진행 상황을 반영한 실적 컬럼은 직접 수기 갱신.

- 클래스 간 관계가 명확한가

**팀/개인 수정 내용:**
> 설계자와 회의 후 `SceneManager`만 신설. 입력 처리는 `Player` 클래스 내부에 잠정 유지하기로 합의.

**최종 활용 결과:**

| 항목 | 내용 |

| 관련 주차 / 활동 | 13주차 — MVP 스펙 문서 작성 |
| 사용 도구 | Claude (웹) |
| 역할 연관성 | PM — 구현 단계 단일 기준 문서 작성. 가장 중요한 PM 산출물 중 하나 |

**프롬프트:**
```
최종 비전 문서(요구사항/클래스/UC 다이어그램)는 너무 크기 때문에
MVP 스펙을 별도로 만들고 싶어. 다음을 명시해줘:
1) MVP에 포함되는 기능
2) MVP에 포함하지 않는 기능(AI에게 시키지 말 것 명시)
3) Definition of Done
4) 기술 스택 확정
2D 횡스크롤, pygame-ce, 1스테이지 클리어가 목표.
```

**AI 응답 요약:**
> MVP 한 줄 정의 → 스코프 IN/OUT → 기술 스택 → DoD 순으로 구조화된 문서 초안을 생성. "AI 프롬프트 시 OUT 목록을 그대로 던지고 추가하지 말 것 명시" 가이드를 포함한 것이 특히 유용했음.

**비판적 검토:**
> AI 초안에는 "사운드는 포함"으로 되어 있었으나, 5인 팀·3주 구현 일정상 비현실적이라 OUT으로 이동. 또한 "멀티 스테이지"를 IN으로 잡고 있었는데 1스테이지로 축소.

**팀/개인 수정 내용:**
> IN/OUT 경계를 팀 회의로 재조정. 특히 "회원가입/로그인 / 인벤토리 / 점수·랭킹 / 세이브·로드 / 적 AI" 등을 명시적으로 OUT 처리해 개발자가 AI에 코드 요청 시 혼동을 줄이도록 함.

**최종 활용 결과:**
> `docs/mvp-spec.md`로 반영. 이후 모든 구현 요청 시 이 문서를 함께 첨부하는 것을 팀 규칙으로 확정.

**프롬프트:**
```
첨부한 MVP 스펙 문서를 기준으로 pygame-ce 프로젝트 베이스라인을 생성해줘.
- src/main.py 진입점
- src/scenes/ (메뉴, 게임, 클리어)

**AI 응답 요약:**
> 디렉터리 구조와 각 모듈의 스켈레톤 코드를 생성. settings.py에 화면 크기·중력·점프 속도 등 상수 분리됨.

**비판적 검토:**
> 초안에 `inventory.py`, `save_manager.py` 등 OUT에 명시된 기능 파일이 포함되어 있었음. "OUT 목록에 있는 파일은 만들지 마"라고 재지시한 후에야 제거됨. AI도 학습 패턴상 게임 프로젝트의 "일반적인" 구조를 자동으로 채우려는 경향이 있어 명시적 차단이 필요했음.

**팀/개인 수정 내용:**
> OUT 파일 제거 후 직접 실행 테스트. import 경로 오류 1건 수정. 실제 게임 로직(점프 곡선, 체크포인트 등)은 개발자에게 인수인계.

**최종 활용 결과:**
> `src/` 디렉터리에 반영(커밋 ce546f9). 개발자가 이 베이스라인 위에서 구현 진행.

| 사용 도구 | Claude Code (CLI) |
| 역할 연관성 | PM — 분석가 산출물(`docs/usecase.md`)에 강의 요구 양식 적용 |

**프롬프트 (입력한 내용 요약):**
```
강의자료(2026-1st 소프트웨어개론_08_객체지향 분석.pdf)에 나와있는 형식으로
유스 케이스 설명서를 작성해줘.
식별부, 정상 시나리오 정의부, 예외 처리 정의부 세 섹션이 필요하다고
강의에 나와있어. 기존 usecase.md에 추가해줘.
```

**AI 응답 요약:**
> 먼저 강의자료 PDF를 pdftotext로 추출하고 Edge 브라우저 스크린샷으로 실제 슬라이드 표 구조를 확인했다. 14페이지(식별부 표), 15페이지(정상 시나리오 단일 컬럼 표), 17페이지(예외 처리 대안 흐름 표)의 형식을 파악한 뒤 UC-01~UC-30 전체 30개 유스 케이스에 세 섹션을 작성했다. 식별부는 "유스 케이스 이름/ID/중요도, 주 액터/유형, 이해관계자, 간략한 설명, 트리거, 관계" 6행 표로, 정상 시나리오는 `정상 이벤트 흐름 (정상 시나리오)` 단일 컬럼 번호 목록으로, 예외 처리는 `대안 및 예외 흐름` 표에 `4a:`, `3b:` 식의 단계 번호 참조 방식으로 작성했다.

**비판적 검토:**
> AI가 처음에는 강의자료를 읽지 않고 일반적인 유스 케이스 양식(트리거 단독 테이블 등)으로 작성했다. 강의자료를 직접 확인하도록 유도한 결과, 식별부의 `이해관계자(Stakeholders)` 항목과 `관계` 항목의 분리 방식(연관/포함/일반화/확장을 한 셀에 슬래시로 구분)이 슬라이드와 다르게 생성되어 직접 수정을 지시했다. 또한 예외 처리의 단계 번호 참조 방식(`4a:`)이 슬라이드 예시(S-1, 2a1 형식)와 약간 달랐으나, 한국어 게임 도메인에서는 더 직관적인 현재 방식이 적합하다고 판단해 유지했다.

**팀/개인 수정 내용:**
> 관계 항목의 포함/확장 관계 일부를 UC 다이어그램과 대조하여 직접 수정(예: UC-11 일시정지는 저장·종료 확장 관계 명시). 중요도(상/중/하) 수치는 PM이 기능 우선순위를 반영해 직접 부여했다.

**최종 활용 결과:**
> `docs/usecase.md`에 UC-01~UC-30 전체 유스 케이스 설명서로 반영.

## 성찰 일지

### 제출 시점: AI 로그 #3 (M3)

**이번 기간 동안 AI를 어떻게 활용했나요?**
> 주로 PM 역할의 **문서 작성·형식 통일·구조화** 작업에 사용. 요구사항 정의서의 docx→md 변환, WBS 표 생성, FP 분류, Mermaid 다이어그램 변환, MVP 스펙 문서 구조화 등 "정형화·가독성 개선" 영역에서 가장 자주 사용했다. 또한 구현 후반에는 Claude Code(CLI)를 통해 pygame-ce 베이스라인 코드의 스켈레톤을 생성했다. 한 학기 동안 약 30~40회 활용했다.

**AI 활용에서 가장 도움이 된 순간은?**
> MVP 스펙 문서(`docs/mvp-spec.md`) 작성 시점. 최종 비전 문서가 비대해져 개발자가 "어디까지가 MVP인가"를 매번 물어보던 문제가 있었는데, Claude에 "OUT 목록을 명시하고 AI에 시키지 말 것"이라는 구조를 제안받아 단일 기준 문서가 완성됐다. 이후 팀원들의 AI 코드 요청 품질이 눈에 띄게 개선됐다.

**AI 활용에서 가장 어렵거나 주의가 필요했던 순간은?**
> 베이스라인 코드 생성 시 OUT으로 명시한 기능(인벤토리, 세이브 등)을 AI가 "관행적으로" 포함시키려 한 점. 한 번의 지시로는 부족했고 "OUT 목록에 있는 파일은 만들지 마"라고 재차 차단해야 했다. AI는 "일반적인 게임 프로젝트는 이런 구조"라는 학습 패턴이 강해, 스코프 통제는 사람의 책임이라는 것을 체감했다.

**AI 없이 직접 해야 했던 작업은 무엇이었나요? 이유는?**
> ① **팀원 간 역할 조율·회의 진행** — AI는 회의 분위기와 개인 일정을 알 수 없음. ② **MVP 스코프 결정** — 무엇을 빼고 무엇을 남길지는 팀 합의 사항이지 AI가 결정할 수 없음. ③ **WBS 실적 컬럼 갱신** — 실제 진행 상황은 PM이 매주 추적해야 정확함. ④ **팀원 산출물에 대한 가치 판단형 코멘트** — "이 부분이 우리 팀 강점이다/약점이다" 같은 평가는 AI 답변보다 본인 책임이 더 명확해야 했다.

**다음 기간에는 AI를 어떻게 다르게 활용할 건가요? (이번 학기 AI 활용에서 얻은 가장 중요한 교훈)**
> 가장 중요한 교훈은 **"AI에게 맡길 일과 사람이 결정할 일을 먼저 분리하라"**는 것. 초반에는 "AI가 잘 만들어주겠지" 하고 통째로 던졌다가 잘못된 분류·과한 구조를 받아오는 비용이 컸다. 후반부에는 ① 입력 형식, ② 출력 형식, ③ 절대 하지 말 것을 미리 못 박은 뒤 요청하는 방식으로 바꾸자 결과 품질이 일정해졌다. 다음 프로젝트에서는 처음부터 "MVP 스펙 + OUT 목록"을 정해두고 AI에 첨부하는 워크플로를 표준화할 계획이다.
